# GlowCast — Latest State (2026-05-26)

> Snapshot of the GlowCast project at the end of the build/release session.

## Overview

**GlowCast** is a native macOS menu-bar app that drives **HyperX QuadCast 2 S** RGB lighting on macOS. The QuadCast 2 S has no macOS software and doesn't store color in firmware — GlowCast keeps a 30 Hz render loop pushing per-LED color packets via `IOHIDManager`, with auto-reconnect on unplug/replug and launch-at-login.

Built from scratch in this session: design spec → implementation plan → 28 commits → public GitHub repo → promo.

## Reverse-Engineered Protocol (the key asset)

- **Device:** VendorID `0x03f0`, ProductID `0x02b5` (QuadCast 2 S controller)
- **RGB collection:** HID vendor collection, usage page `0xff13` / usage `0xff00`, on interface 1. It is a **non-primary** usage on macOS — match on VID/PID and filter on the usage pair, otherwise `IOHIDManager` misses it.
- **Transport:** `IOHIDDeviceSetReport(.output, reportID: 0, …)`, 64-byte reports. **Write-only works** — no need to drain input reports.
- **Refresh sequence** (every frame, ~30 Hz):
  1. Header packet: `0x44, 0x01, 0x06, 0x00, …zeros` (64 bytes)
  2. Six data packets: `0x44, 0x02, <index 0..5>, 0x00, <RGB triples>` (64 bytes each)
- **LEDs:** 108 total. 20 RGB triples per packet (bytes 4..63); packet 5 carries 8 triples then zeros.
- **Color byte order:** R, G, B. Cyan `#06b6d4` = `06 b6 d4`.
- **Live-driven** — no firmware persistence. Stop driving → LEDs revert to default.

## macOS Permissions

`IOHIDDeviceOpen` returns `kIOReturnNotPermitted` until **Input Monitoring** is granted (because the interface also exposes a mouse collection alongside the vendor RGB one).

To make the grant **persist across rebuilds**, the app is code-signed with a **stable self-signed cert**:

```
Authority: GlowCast Self Signed
```

The cert was created via `openssl` + `security import` into the login keychain. TCC binds the grant to the **cert identity** (not the cdhash), so rebuilds keep the grant.

Grant location: System Settings → Privacy & Security → **Input Monitoring** → **GlowCast** enabled.

**Title**

```
PSA for Mac users with a QuadCast 2 S — how to actually get the RGB working (NGENUITY is Windows-only)
```

**Body**

```markdown
Sharing this because I spent way too long digging into it and figured the next person could use the heads-up.

If you switched from Windows to Mac with a QuadCast 2 S and noticed your RGB does nothing — that's not a bug. NGENUITY is Windows-only, and even after you set a color there, the 2 S resets to default the moment it's plugged into a Mac.

What I learned:

- The 2 S is **live-driven** — it doesn't store color in firmware. Something on the host has to keep sending packets, otherwise the LEDs go default. That's why NGENUITY colors don't survive the switch.
- The existing Linux/Win CLI `quadcastrgb` (by Ors1mer) uses libusb, which macOS blocks: `IOHIDFamily` holds the HID interface exclusively → `Access denied`.
- macOS's **IOHIDManager** does work for it. The RGB collection sits on a *non-primary* HID usage (page `0xff13` / usage `0xff00`, on the controller's interface 1) — so any tool matching only primary usages misses it.

I ended up writing a small open-source SwiftUI menu-bar app to scratch this itch (link in the comments — respecting the no-self-promo rule). Auto-reconnects on unplug, 30 colors + 10 modes, runs at login. MIT licensed.

Anyone else here found other working tools for the 2 S on Mac, or has luck with the regular QuadCast 2?
```

## Future / v2 Ideas

- **Independent upper / lower zone colors** (the device supports it).
- **Optional system-audio reactivity** via ScreenCaptureKit / process-tap APIs (the microphone-input version was removed at user request this session).

- **Install to `/Applications`** for the rock-solid "Launch at Login" path. Currently runs from `build/`. The stable cert means the Input Monitoring grant would carry over to the new path.
